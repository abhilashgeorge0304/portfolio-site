### The Script Nobody Knew Had Failed

Our team ran dozens of automated scripts on a virtual machine — processing alerts from connected lighting systems, ingesting data from enterprise resource platforms, syncing records across cloud warehouses. These scripts were the backbone of daily operations across multiple business-critical systems. And we had no idea when they failed.

The only way anyone found out was when a user raised a ticket. By then, the damage was done — stale data, missed alerts, frustrated stakeholders asking why something hadn't updated. We were running a reactive operation and calling it good enough.

Meanwhile, the organisation had a Datadog subscription being used almost exclusively for monitoring product-level alerts from IoT lighting infrastructure. A capable observability platform, barely utilised for anything operational. I saw the gap, and I decided to fix it.

### Understanding What Runs on the VM

Before building any dashboard, I needed to understand exactly what was running on this machine and how different each workload was. It turned out we had two fundamentally different types of automation sharing the same VM — each with its own architecture, failure modes, and business impact.

**Workload 1: Alert Processing Pipeline.** A modular Python application that monitors multiple connected lighting system platforms. The architecture is clean: a main orchestrator initialises configuration, sets up the runtime environment, and triggers operations for each monitored platform in parallel. Each platform has its own processor class — a specialised adaptor that fetches alert emails from an inbox, parses the HTML content, extracts structured alert data, converts it to JSON, and pushes events to the monitoring platform. Some platforms add an uptime calculation layer on top: tracking connected device counts, computing failure thresholds, and only generating alerts when uptime drops below a target — say, below 95%.

The codebase is well-structured. A `core/` layer handles shared utilities, database operations (SQLite with an ORM), and a custom logging module that sends execution summaries via email. A `services/` layer wraps external integrations — email APIs, the monitoring platform's API. `fetchers/` and `exporters/` handle the data flow in and out. And `processors/` contains the per-platform business logic. The whole thing runs from a single entry point that orchestrates all processors concurrently.

**Workload 2: Data Ingestion Pipeline.** A completely different beast. An ERP system uploads CSV files to cloud storage every 30 minutes. A scheduled Python script on the VM downloads these files, transforms them from a Windows-specific encoding to a format compatible with the cloud data warehouse, and loads them into target tables. This pipeline has its own failure modes — encoding mismatches, missing files, malformed CSVs, cloud storage connectivity issues — none of which overlap with the alert processing pipeline.

Two pipelines. Different architectures. Different schedules. Different owners on the team. Different failure modes. One VM. Zero visibility into any of it.

### Building the Foundation

I started with what I could build and validate without depending on anyone else.

**Step 1: Agent installation.** I installed the Datadog agent directly on the VM, configured the API keys, established connectivity, and validated that system-level metrics — CPU, memory, disk usage, uptime — were flowing into the platform correctly. This was the foundation everything else would build on.

**Step 2: Proof of concept dashboard.** With the agent streaming data, I built a simple sanity-check dashboard: VM health metrics displayed in real time, confirming the monitoring approach was viable. This POC served a critical purpose — it was what I used to get stakeholder buy-in for the full project. Showing a working dashboard with live metrics is more convincing than any slide deck.

**Step 3: Code-level instrumentation.** VM metrics tell you the machine is alive. They don't tell you whether the scripts running on it are succeeding or failing. The next layer required instrumenting the code itself — emitting metrics on execution status, duration, and errors using the DogStatsD API so that application-level behaviour was visible alongside infrastructure health.

I defined the metric schema and the monitoring specification — what metrics to emit, at what granularity, with which tags — and delegated the code instrumentation work to team members who had deeper familiarity with each pipeline's codebase. One team member handled the alert processing pipeline instrumentation, another handled scaling it across all monitored systems. I validated the integration end-to-end: making sure what the code emitted matched what the dashboard expected.

### Designing Two Dashboards for Two Audiences

This is where the project stopped being a technical exercise and became a product decision. We had two groups who needed visibility into the same VM — but they needed to see fundamentally different things.

**The business stakeholders** didn't care about CPU percentages or Python stack traces. They cared about: Are the lighting systems being monitored? Is the data pipeline running on time? Are there any failures I need to know about before someone complains? Their dashboard needed to be glanceable — a quick check that says "everything is fine" or "something needs attention."

**The digital/engineering team** needed the opposite: granular detail. Which specific processor failed? At which step — fetching, parsing, transformation, or export? Was it a code error or an infrastructure problem? They needed a debugging surface, not a status page. And they needed it fast — because every minute spent figuring out *where* something failed is a minute of escalation risk.

I designed both dashboards to pull from the same underlying metric stream — no data duplication, just different views of the same truth.

### The Business Dashboard

The business dashboard is designed to be read in under ten seconds. At the top: two large status indicators — one for the alert processing pipeline, one for the data ingestion pipeline. Green means everything ran successfully on the last execution. Anything else means someone should ask a question.

Below that, a **job success rate timeseries** showing the percentage of successful runs over the last 24 hours. Business stakeholders don't need to know which specific processor failed — they need to know whether the overall success rate is trending down. A sustained dip from 99% to 92% tells them something systemic is happening, even if each individual failure seems minor.

A **system status grid** shows the last-run result for each monitored platform and the ingestion pipeline. Green checkmarks for healthy, amber warnings for delayed runs, red crosses for failures. This is the widget that gets checked first thing in the morning and last thing before end of day.

At the bottom: **VM uptime** (how long since the last restart — a proxy for stability) and an **active alerts count** with a one-line description of what's currently flagged. If this number is zero, the dashboard is doing its job. If it's not, the description tells the stakeholder exactly what's wrong without needing to open a ticket.

### The Engineering Dashboard

The engineering dashboard assumes the reader already knows something is wrong and needs to find out *where* and *why*.

**VM resource metrics** — CPU and memory timeseries — sit at the top. These aren't just decoration. When a script fails, the first question is always: "Is it a code problem or an infrastructure problem?" If CPU was at 98% when the script failed, it's a resource issue. If CPU was at 30%, it's in the code. Having these metrics right next to the execution log eliminates the first five minutes of every debugging session.

The **job execution panel** is the centrepiece. It shows each processor's execution history as a horizontal bar — green segments for successful runs, red segments for failures. You can see at a glance not just that something failed, but *when* it failed and whether it recovered. Below the bars, a one-line error summary points to the specific failure: which processor, which step (fetch, parse, transform, export), and what the error was.

An **error log stream** shows the most recent log entries filtered to warnings and errors only. This is where the sub-job granularity pays off. Instead of "alert_pipeline failed," the log shows "system_b: OutlookFetcher timeout at 14:32" — immediately isolating the root cause to a specific platform's email fetch operation, not the entire pipeline.

**Disk usage** and **average execution duration** round out the panel. Disk matters because the VM accumulates processed files, archived emails, and database records over time — if nobody's watching, the disk fills up and everything fails at once. Execution duration matters because gradual slowdowns are the early warning of bigger problems — a processor that used to take 2 minutes and now takes 8 is probably hitting an API rate limit or a growing dataset.

### Sub-Job Granularity: The Design Decision That Made It Useful

The most important design decision wasn't what to monitor — it was **at what level of depth**. The easy approach would have been to track whether each main script ran or didn't. But that tells you almost nothing useful when something breaks.

Take the alert processing pipeline. The orchestrator runs all processors in parallel. If one processor fails, the orchestrator still completes — the other processors finish successfully. Without sub-job monitoring, the dashboard would show "pipeline: success" even though one platform's alerts went completely unprocessed.

I designed the monitoring to capture the processor hierarchy. Each processor reports its own execution status, duration, and error count independently, tagged by platform and by processing step. The dashboard then visualises this as a per-processor timeline — so when something breaks, you can see that the main orchestrator is fine, platform A is fine, but platform B's HTML parser threw an exception at 14:32 because the email format changed.

When the pipeline fails, the dashboard shows whether it was a connectivity issue to cloud storage, an encoding problem in the CSV, or a schema mismatch at the warehouse. Different failures, different fixes, immediately visible.

### Delivering Through Structured Project Management

Building the dashboard was a technical exercise. Scaling it across all systems and making it production-grade was a project management exercise. I structured the work using phased delivery — clear milestones, defined ownership, and dependencies mapped upfront.

**Phase 1 — Infrastructure (owner: me).** Agent installation, connectivity validation, POC dashboard. This had to be complete before anything else could start. No dependencies on anyone else.

**Phase 2 — Single application instrumentation (delegated + validated).** Implemented code-level monitoring in one representative application. I defined the metric schema and specifications. A team member with deeper code familiarity handled the DogStatsD integration — emitting metrics for execution status, duration, and errors. I validated the end-to-end flow: metrics appearing in Datadog, dashboard widgets responding correctly.

**Phase 3 — Log monitoring integration (coordinated handoff).** Configured Datadog log pipelines and filters. Created log-based dashboard widgets. The log schema had to match what the dashboard expected — which meant the person building the log integration and the person designing the dashboard needed a shared specification. I wrote that spec and managed the handoff.

**Phase 4 — Scale to all systems (template + review).** Rolled the monitoring code out to all processors and the ingestion pipeline. Each system's scripts were managed by a different team member, so I created a monitoring instrumentation template that standardised how every application reports. Team members adapted the template to their pipelines; I reviewed each implementation and validated the dashboard reflected all systems correctly.

**Phase 5 — Alerting (simulated failures).** Configured Datadog monitors to trigger email alerts on job failures, abnormally long execution times, error spikes, and resource exhaustion. Validated through simulated failure scenarios — intentionally breaking scripts to confirm alerts fired correctly and reached the right people.

Each phase had a dependency on the previous one — you can't build log-based widgets without log pipelines, and you can't configure meaningful alerts without the full metric stream flowing. This dependency chain is why structured phasing mattered. Without it, people would have been building dashboards for metrics that weren't being emitted yet.

### Going Further: Predictive Resource Forecasting

The dashboard solved the visibility problem. But even with monitoring in place, we were still reacting to resource exhaustion after the fact — scripts failing because the VM ran out of disk space during a heavy processing cycle, or CPU spiking during concurrent runs.

I'm now extending the dashboard with a **predictive forecasting layer** — using Datadog's built-in forecast functions alongside a unit economics model I'm building on top. The idea: if you know how much resource one monitored system consumes on average, you can project when the VM will need an upgrade based on growth. Instead of discovering disk is full at 2 AM when a script fails, the dashboard would alert days or weeks in advance — before resources become a problem.

The goal is to make the system robust enough to work even after team members rotate. The dashboards, the alerts, and eventually the forecasts should be self-sustaining — not dependent on any single person knowing the current state of the VM. The system knows.

### What This Changed

Before this project, we had no visibility into operational workflow health. Failures were discovered by users raising tickets. Debugging meant logging into the VM and reading local log files. Nobody could tell business stakeholders whether systems were healthy without checking manually.

After: every script, every sub-job, every failure across two fundamentally different workloads is visible in real time — across two purpose-built dashboards designed for two distinct audiences. Business stakeholders check the status grid and success rate. The engineering team debugs from error logs and resource metrics. Alerts fire before users notice anything. Escalation volume dropped by approximately 40% — leadership now has direct visibility into team output without having to ask.

The project was as much a project management exercise as a technical one. From the solo POC through delegated instrumentation, standardised templates, coordinated handoffs, and phased rollout — the PM discipline is what made this scale beyond a single person's monitoring script into a team-wide operational capability.
