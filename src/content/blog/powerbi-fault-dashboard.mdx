### One Dashboard, Many Languages

Our organisation needed a fault reporting dashboard for lighting systems — a central view showing which systems were reporting faults, how frequently, and across which regions. The data existed. The problem was that it arrived in multiple languages.

Reports covered installations across different countries, and fault descriptions, status fields, and notes were all recorded in the local language. A French installation would report *"aucun défaut signalé"* while a German one would log *"kein Fehler gemeldet"* — both meaning the same thing, but looking completely different to any aggregation logic or visual filter.

The straightforward approach — maintaining manually translated lookup tables — would have been fragile and unsustainable. Every time a new fault description appeared in the data, someone would need to notice, translate it, and update the lookup. That's exactly the kind of invisible maintenance that gets neglected until a dashboard starts showing untranslated rows and stakeholders lose trust in the data.

I built something different: a translation layer that runs automatically inside Power BI's data refresh pipeline, translates what it hasn't seen before, caches everything it has, and normalises known phrases to canonical forms. Adding a new language requires no manual intervention — the system handles it on the next refresh.

### The Dashboard

I gathered requirements from business stakeholders on what metrics to display: fault frequency by installation, fault type distribution, time-to-resolution patterns, and regional comparisons. The dashboard uses DAX measures for calculated KPIs — rolling fault rates, comparison against baselines, and conditional formatting to highlight installations that are trending worse over time.

The data loads from Excel sources through Power BI's transformation pipeline, where the translation layer sits as a Python execution step before the data reaches the model.

### The Translation Engine

The translation runs as a Python script inside Power BI's data transformation step — sitting between raw data ingestion and the analytics model. The design priorities were: **translate only what's new** (don't re-translate cached values), **normalise known phrases** (so variants of the same meaning map to one canonical form), and **fail gracefully** (if translation fails, pass the original value through rather than breaking the refresh).

**Persistent caching** was the key design choice. The translation cache is stored as a JSON file on disk, structured by language and by translation type (canonical phrases vs. on-the-fly translations). On each Power BI refresh, the script loads the cache first and only calls the translation API for values it hasn't seen before. This means the first refresh for a new language is slow (every value needs translating), but subsequent refreshes are near-instant — hitting the cache for 95%+ of values.

**Synonym normalisation** handles the reality that the same concept appears in multiple forms. Phrases like "no fault reported," "no faults reported," and minor spelling variations are all mapped to a single canonical form before translation. This prevents the cache from storing five slightly different translations of the same concept, and ensures that dashboard filters work consistently — a user filtering by "No fault reported" sees all the records, not just the ones that happened to use that exact phrasing.

**Graceful failure** means that if the translation API is unreachable or throws an error for a specific value, the original text passes through unchanged. The dashboard still works — it just shows some untranslated values until the next successful refresh. No broken visuals, no empty cells.

### What This Enabled

The dashboard now supports multiple languages without any manual translation maintenance. When data from a new region appears, the translation layer detects the new language on the next refresh and translates automatically. The cache builds incrementally — within a few refresh cycles, virtually every value is cached and the translation step adds negligible time to the pipeline.

For stakeholders, the experience is seamless: fault data from any region appears in the dashboard in a consistent language, filterable and comparable. The engineering work that makes this possible is invisible — which, for infrastructure like this, is exactly the point.
