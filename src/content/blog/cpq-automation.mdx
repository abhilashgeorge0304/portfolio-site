### The Work Nobody Wants to Do Twice

Enterprise configuration platforms generate a particular kind of operational pain: the work that isn't intellectually hard but is tedious, repetitive, and brutally error-prone. Copying files. Renaming them to match naming conventions. Updating dynamic links. Deploying to sync servers. Triggering builds. Verifying outputs. Doing it again for the next sales organisation. And again for the next language.

Our CPQ platform had exactly this problem — across two separate workstreams that I encountered within weeks of each other. In both cases, the same instinct kicked in: this is a scripting problem, not a staffing problem.

### Project 1: Template Deployment Automation

The business needed new output document templates for a product line expansion. In CPQ, output documents are the customer-facing deliverables — quotes, sales agreements, pricing sheets, warranty terms, disclaimers. Each document type is a template file that the platform populates dynamically based on the configuration.

The complexity wasn't in any single template. It was in the **structure**.

Each template file uses **dynamic linking** — file paths contain variables like `{BookletLanguage}` and `{SalesArea/SalesOrganization}` that resolve at generation time. This means every file has to be named *exactly right* and placed in *exactly the right directory* for the links to resolve. One typo in a filename and the entire document export fails silently — you only discover it when a sales rep tries to generate a quote and gets a broken PDF.

The template set included over twelve document types: quote headers, reference letters, quote tables, pricing and delivery terms, sales agreements (with separate partner and dealer variants), extended warranties, service terms, disclaimers, and more. Each document type needed to exist for every sales organisation, and eventually for every language. The estimation was roughly 2-3 hours per additional sales organisation (copying and renaming all files, verifying links) and approximately 20 hours per new language (full translation cycle).

**The manual process** for deploying to a new sales organisation:
1. Copy all ~12 template files
2. Rename each one to match the target sales org naming convention
3. Verify every dynamic link resolves correctly with the new names
4. Connect to the extraction server
5. Upload all files to the sync folders
6. Trigger the sync via the build server
7. Export a test document to verify everything works
8. Repeat for each additional sales organisation

I gathered the business requirements, scoped the work, defined story points for each document type based on complexity (some were straightforward copies, others like the quote table involved 8-9 hours of formatting work), and split the implementation into manageable tasks.

Then I wrote **a script that automates the entire copy-rename-link-verify pipeline end to end**. What previously took roughly two weeks of careful manual work now runs in about an hour — with fewer errors, because the script doesn't make typos in filenames.

### Project 2: Approval Flow Automation

A separate workstream required setting up a new approval flow for a regional market — including creating new user roles, decommissioning old ones, and carefully preserving shared roles that were used across multiple regions.

The manual approach was to configure each approval rule individually through the platform's admin interface — a process that takes days when dozens of rules need to be created, tested, and validated. The risk wasn't just time — it was accidentally deleting or modifying a shared role that would break approval flows in other regions.

I wrote a script to automate the rule creation, cutting days of manual configuration to a few hours. The script also generated documentation for the new setup — ensuring that the approval flow changes were auditable and that the next person who needed to modify them wouldn't have to reverse-engineer the configuration from the UI.

### The Pattern

These weren't isolated wins. They were two instances of the same problem pattern: enterprise platforms with configuration complexity that grows multiplicatively (more templates × more languages × more sales orgs; more regions × more roles × more rules), where the manual approach doesn't just scale linearly — it scales with compounding error risk.

The instinct in both cases was the same: understand the structure, map the repetitive steps, and script the problem away. The automation isn't just faster — it's *more reliable*, because the script executes the exact same steps in the exact same order every time. That reliability matters more than the time savings, because a single broken template or misconfigured role can cascade into hours of debugging that nobody budgeted for.
